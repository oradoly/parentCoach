# 07. Decision Log

이 문서는 확정된 제품·기술 결정을 기록한다. Codex는 아래 결정을 임의로 번복하지 않는다. 변경이 필요하면 새 결정 항목에 이유, 대안, 영향을 기록하고 사용자의 명시적 승인을 받는다.

## D-001. 주 사용자는 부모다

- 상태: 승인
- 결정: 앱의 주 사용자는 초등 5~6학년 자녀를 둔 부모·보호자다.
- 결과: 화면, 카피, 정보 구조는 부모 관점으로 설계한다.
- 제외: 아이가 직접 자유 대화하며 답을 받는 학생용 튜터 경험.

## D-002. MVP는 수학만 지원한다

- 상태: 승인
- 결정: 첫 버전은 대한민국 초등학교 5~6학년 수학에 한정한다.
- 이유: 정확성, 교육 적합성, 평가 가능성을 먼저 확보하기 위해서다.
- 결과: 다른 과목 UI나 범용 프롬프트를 미리 구현하지 않는다.

## D-003. 핵심 순서는 역질문 → 힌트 → 최종 풀이 → 비슷한 문제다

- 상태: 승인
- 결정: 최종 답보다 부모 코칭 흐름을 우선한다.
- 결과: 첫 결과에 정답을 노출하지 않고 힌트는 단계적으로 연다.

## D-004. 문제 인식은 사용자 확인을 거친다

- 상태: 승인
- 결정: OCR·비전 결과를 자동으로 확정하지 않는다.
- 결과: 수정 가능한 확인 화면을 필수로 둔다.

## D-005. MVP는 회원가입 없이 핵심 기능을 쓸 수 있다

- 상태: 승인
- 결정: 아이 계정과 장기 프로필을 요구하지 않는다.
- 결과: 임시 세션과 짧은 데이터 보관을 기본으로 한다.

## D-006. AI 출력은 구조화 스키마로 받는다

- 상태: 승인
- 결정: 자유 형식 응답을 UI에서 직접 파싱하지 않는다.
- 결과: Recognition과 Coaching에 버전 있는 JSON Schema를 사용한다.

## D-007. LLM 답변은 가능한 범위에서 별도 검증한다

- 상태: 승인
- 결정: 산술·분수·단위 등은 결정적 검산기를 둔다.
- 결과: 생성 결과와 검산이 충돌하면 정답을 노출하지 않는다.

## D-008. 권장 초기 구현은 Expo + TypeScript 서버다

- 상태: 제안 승인 기준
- 결정: 모바일은 Expo React Native, 서버와 공유 계약은 TypeScript, 패키지 관리는 pnpm workspace를 기본으로 한다.
- 이유: 하나의 언어로 모바일·API·계약을 빠르게 구축하기 위해서다.
- 변경 조건: 팀 역량, 배포 환경, 기존 코드베이스가 명확히 다른 선택을 요구할 때 ADR과 사용자 승인을 거친다.

## D-009. OpenAI 호출은 서버에서 Responses API로 한다

- 상태: 제안 승인 기준
- 결정: 이미지 입력과 구조화 출력을 서버 측 OpenAI Responses API로 구현한다.
- 결과: 모델 이름은 환경 설정으로 분리하고 클라이언트에 API 키를 넣지 않는다.

## D-010. 문제 사진은 임시 데이터다

- 상태: 승인
- 결정: 원본 사진을 학습 기록처럼 영구 저장하지 않는다.
- 결과: 처리 후 즉시 삭제 또는 짧은 TTL, 위치 메타데이터 제거, 로그 미포함을 기본으로 한다.
