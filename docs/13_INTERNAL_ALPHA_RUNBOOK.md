# 13. Internal Alpha Runbook

이 문서는 M8-4 내부 알파 1차 운영 준비용 runbook이다. 목적은 실제 부모 테스트를 공개 베타처럼 운영하는 것이 아니라, 샘플 앱을 핸드폰에서 먼저 확인하고 부모 3~5회 파일럿을 안전하게 진행할 수 있게 하는 것이다.

## 1. 운영 판정

- 현재 판정: **내부 알파 제한 진행**
- 가능: 내부 팀 시연, 샘플 앱 폰 smoke, 부모 3~5회 파일럿
- 불가: 공개 베타 오픈, 지원 범위 확장, 실제 부모 테스트 완료로 표시

M8 closeout 근거는 `docs/12_M8_CLOSEOUT.md`를 따른다.

## 2. 진행자 준비물

- Mac과 테스트 핸드폰이 같은 Wi-Fi에 연결되어 있어야 한다.
- 테스트 핸드폰에 Expo Go 또는 개발 빌드 실행 환경이 준비되어 있어야 한다.
- API 서버와 모바일 앱을 같은 저장소 버전에서 실행한다.
- 문제 사진은 초등 5~6학년 수학 한 문제만 보이게 준비한다.
- 테스트 전 참여자에게 `docs/10_PRIVACY_AND_BETA_NOTICE.md`의 핵심 안내를 설명한다.
- 테스트 1회 관찰 기록은 `docs/14_INTERNAL_ALPHA_OBSERVATION_TEMPLATE.md`를 사용한다.
- 3~5회 파일럿 집계와 계속 진행/수정 후 진행/중단 판정은 `docs/15_INTERNAL_ALPHA_PILOT_LOG.md`에 남긴다.

## 3. 기록 금지 항목

아래 정보는 관찰 노트, 이슈, 채팅, 스크린샷 파일명에 남기지 않는다.

- 아이 이름, 학교, 학번, 얼굴, 연락처
- 원본 문제 이미지 또는 image data URL
- 문제 전문
- 프롬프트 전문
- 원본 AI 응답 전문
- 성적, 진단, 장기 학습 평가

필요하면 문제는 `domain`, `skill`, 오류 단계, requestId로만 식별한다.

## 4. 로컬 API 준비

Mac의 LAN IP를 확인한다.

```bash
ipconfig getifaddr en0
```

값이 비어 있으면 다른 네트워크 인터페이스를 확인한다.

```bash
ifconfig | grep "inet "
```

모델 키 없이 샘플 앱 흐름만 확인할 때는 local fixture mode를 사용한다. 이 모드는 실제 사진 인식이 아니다. 어떤 사진을 올려도 같은 샘플 문제와 코칭이 반환되므로, 홈부터 피드백까지의 surface와 답 노출 순서만 확인한다.

```bash
ENABLE_LOCAL_AI_FIXTURES=true \
NODE_ENV=development \
PORT=3001 \
ALLOWED_WEB_ORIGINS=http://127.0.0.1:8081,http://127.0.0.1:4173 \
pnpm --filter @parent-coach/api start
```

실제 OpenAI 호출로 사진별 인식과 코칭을 확인할 때는 `OPENAI_API_KEY`와 모델 환경 변수를 서버 환경에만 둔다. 클라이언트에는 비밀키를 넣지 않는다. 이때는 `ENABLE_LOCAL_AI_FIXTURES`를 설정하지 않고 API를 시작한다.

```bash
NODE_ENV=development \
PORT=3001 \
ALLOWED_WEB_ORIGINS=http://127.0.0.1:8081,http://127.0.0.1:4173 \
pnpm --filter @parent-coach/api start
```

실제 AI mode에서 키가 없거나 모델 호출이 비활성화되어 있으면 앱은 임의 풀이를 보여 주지 않고 모델/API 사용 불가 상태로 멈춰야 한다.

## 5. 핸드폰 앱 실행

다른 터미널에서 Mac LAN IP를 사용해 Expo를 시작한다.

```bash
EXPO_PUBLIC_API_BASE_URL=http://<MAC_LAN_IP>:3001 \
pnpm --filter @parent-coach/mobile start
```

M9-1 이후 static web surface로 반복 확인할 때는 아래 helper를 먼저 실행한다. API 서버와 static server가 이미 떠 있으면 `--skip-export`로 빠르게 health, CORS, bundle URL만 확인할 수 있다.

```bash
pnpm phone:smoke
pnpm phone:smoke -- --skip-export
```

helper가 출력하는 `Phone URL`을 핸드폰 브라우저에서 연다. static server가 꺼져 있으면 helper 출력의 `python3 -m http.server ...` 명령을 별도 터미널에서 실행한다. API 서버가 꺼져 있고 샘플 흐름만 확인한다면 helper 출력의 `ENABLE_LOCAL_AI_FIXTURES=true ...` 명령을 별도 터미널에서 실행한다. 실제 사진별 AI 확인을 하려면 위 실제 AI mode 명령으로 API를 다시 시작한다.

핸드폰에서 QR을 열고 앱이 뜨면 다음을 먼저 확인한다.

- 홈 제목이 부모에게 말하는 문장인지 확인한다.
- 촬영 또는 사진 선택 버튼이 보이는지 확인한다.
- 업로드 안내에 개인정보 제외 문구가 보이는지 확인한다.

핸드폰에서 API 연결이 안 되면 다음을 확인한다.

- Mac과 핸드폰이 같은 Wi-Fi인지 확인한다.
- `EXPO_PUBLIC_API_BASE_URL`이 `127.0.0.1`이 아니라 Mac LAN IP인지 확인한다.
- Mac 방화벽 또는 VPN이 로컬 포트 접근을 막고 있지 않은지 확인한다.
- API 서버 터미널에 요청 로그가 찍히는지 확인한다.

## 6. 폰 smoke 절차

폰 smoke는 실제 부모 테스트가 아니다. 진행자가 직접 한 번 앱을 만져 보며 핵심 흐름과 터치감을 확인한다.

1. 홈에서 `사진에서 가져오기` 또는 `수학 문제 찍기`를 누른다.
2. 문제 하나만 보이는 사진을 선택하거나 촬영한다.
3. 업로드 전 미리보기와 개인정보 안내가 보이는지 확인한다.
4. 임시 업로드를 시작한다.
5. 인식 결과 확인 화면에서 문제 문장을 확인한다.
6. `맞아요`를 누르기 전에는 코칭이 나오지 않는지 확인한다.
7. `맞아요`를 누르고 부모용 빠른 이해와 첫 역질문을 확인한다.
8. 힌트 1과 힌트 2를 열어 최종 답이 보이지 않는지 확인한다.
9. 힌트 3을 열고, 최종 풀이 버튼이 명시적 행동으로 남아 있는지 확인한다.
10. 최종 풀이를 열고 검산 상태를 확인한다.
11. 비슷한 문제의 답이 기본 접힘인지 확인한다.
12. 피드백 선택지를 하나 누르고 저장 문구를 확인한다.

폰 smoke 중 하나라도 아래에 해당하면 부모 파일럿으로 넘어가지 않는다.

- 인식 확인 전 코칭이 나온다.
- 첫 코칭 화면이나 힌트 1·2에 최종 답이 보인다.
- 오류 상태에서 임의 풀이가 보인다.
- 문제 이미지나 문제 전문이 운영 로그에 직접 출력된다.
- 앱이 핸드폰에서 업로드 이후 진행되지 않는다.

## 7. 부모 파일럿 진행 순서

부모 파일럿은 1차로 3~5회만 진행한다. 목적은 제품 가설을 배우는 것이지 성능을 홍보하는 것이 아니다.

1. 참여자에게 내부 알파이며 공개 제품이 아니라고 설명한다.
2. 이름, 학교, 얼굴, 연락처가 사진에 들어가지 않게 안내한다.
3. 아이가 실제로 물어본 문제 1개 또는 준비된 자체 문제 1개로 시작한다.
4. 진행자는 앱 사용법을 과도하게 설명하지 않고, 막히는 지점을 관찰한다.
5. 촬영 시작 시각과 첫 역질문 확인 시각을 기록한다.
6. 부모가 첫 역질문을 그대로 말할 수 있는지 확인한다.
7. 부모가 힌트를 몇 단계까지 여는지 기록한다.
8. 최종 풀이와 비슷한 문제를 열었는지 기록한다.
9. 마지막에 앱의 피드백 선택지를 하나 고르게 한다.
10. 관찰자는 문제 전문 없이 UX/문구/오류 원인만 기록한다.

## 8. 중단 조건

아래 중 하나라도 발생하면 테스트를 중단하고 `docs/14_INTERNAL_ALPHA_OBSERVATION_TEMPLATE.md`의 P0 항목으로 기록한다.

- 아이 식별 정보가 사진이나 기록에 포함됐다.
- 부모가 앱을 아이에게 직접 답 확인 도구로 넘겨 사용한다.
- 앱이 인식 확인 없이 코칭을 보여 준다.
- 1·2단계 힌트에 최종 답이 드러난다.
- 검산 실패, 모델 비활성화, 지원 불가 상태에서 풀이가 노출된다.
- 참여자가 불편함을 표현하거나 중단을 요청한다.

## 9. 세션 후 정리

테스트 직후 다음만 남긴다.

- 테스트 ID
- 날짜
- 학년대
- domain/skill
- 소요 시간
- 수정 여부
- 열람 단계
- 피드백 선택지
- 오류 단계와 requestId
- 문제 전문 없는 관찰 메모

다음은 남기지 않는다.

- 문제 사진
- 문제 전문
- 아이 식별 정보
- 참여자의 자유 발화 전문
- 모델 응답 전문

3~5회 파일럿을 모은 뒤에는 `docs/15_INTERNAL_ALPHA_PILOT_LOG.md`의 요약 표와 판정 기준으로 다음 단계를 결정한다.

## 10. 1차 파일럿 완료 후 판정

3~5회 파일럿 후 다음 중 하나로 판정한다.

- **계속 진행:** P0 중단 조건이 없고, 부모가 첫 역질문을 대체로 말할 수 있다.
- **수정 후 진행:** 답 누출은 없지만 문구, 촬영, 인식 수정, 오류 안내에서 반복 문제가 있다.
- **중단:** 개인정보, 답 누출, 인식 확인 생략, 검산 실패 노출 중 하나가 발생한다.

이 판정은 공개 베타 승인으로 보지 않는다.
