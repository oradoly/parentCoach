# 02. UX and Content Specification

## 1. UX 목표

부모가 아이 옆에서 오래 화면을 들여다보지 않아도 되도록, **짧게 읽고 바로 말할 수 있는 코칭 카드**를 제공한다.

앱의 기본 경험은 “답 확인”이 아니라 다음 루프다.

```text
부모가 이해한다
→ 아이에게 질문한다
→ 아이 반응을 본다
→ 필요한 만큼만 힌트를 연다
→ 마지막에 풀이를 확인한다
```

## 2. 핵심 디자인 원칙

- 화면의 주어는 “아이”보다 “부모님”이다.
- 각 화면에는 가장 중요한 행동 하나를 우선한다.
- 정답은 시각적으로 첫 화면의 중심이 되지 않는다.
- 코칭 문장은 소리 내어 읽어도 어색하지 않아야 한다.
- 긴 설명보다 접힌 세부 설명을 사용한다.
- 사용자가 급한 상황을 전제로 불필요한 설정을 요구하지 않는다.
- 기본적으로 소리를 재생하지 않는다.

## 3. 주요 화면 흐름

### S01. 홈

목적: 촬영까지 한 번의 명확한 행동만 제시한다.

권장 문구:

- 제목: `아이에게 어떻게 설명할지 같이 준비해 볼까요?`
- 주 버튼: `수학 문제 찍기`
- 보조 버튼: `사진에서 가져오기`
- 안내: `초등 5~6학년 수학 · 한 번에 한 문제`

피해야 할 문구:

- `정답 바로 찾기`
- `숙제 대신 풀기`
- `AI에게 답 받기`

### S02–S03. 촬영과 자르기

가이드:

- 문제 전체와 필요한 그림·보기가 보이게 촬영한다.
- 이름, 학교, 다른 사람 얼굴은 가능하면 제외한다.
- 여러 문제가 있으면 하나만 자른다.

권장 문구:

- `문제 하나만 네모 안에 넣어 주세요.`
- `그림이나 표도 문제의 일부라면 함께 넣어 주세요.`

### S04. 인식 중

단계 문구를 짧게 보여 준다.

1. `문제를 읽고 있어요.`
2. `수식과 조건을 확인하고 있어요.`

AI가 실제로 하지 않은 상세 작업을 과장하지 않는다.

### S05. 인식 결과 확인

제목: `제가 이렇게 읽었어요`

구성:

- 인식된 문제
- 낮은 신뢰도 구간 표시
- `맞아요` 버튼
- `직접 고칠게요` 버튼
- `다시 찍기` 링크

사용자가 수정한 내용이 최종 기준임을 명시한다.

### S06. 부모용 빠른 이해

첫 카드 구성:

- `이 문제의 핵심`: 한 문장
- `구해야 하는 것`: 한 문장
- `설명 방향`: 최대 2문장
- `주의할 점`: 한 문장

정답과 최종 계산 결과는 숨긴다.

예시:

> **이 문제의 핵심** 3/4L 안에 1/8L가 몇 번 들어가는지 알아보는 분수의 나눗셈 문제예요.  
> **설명 방향** “전체 양 안에 한 컵의 양이 몇 번 들어갈까?”라고 바꿔 생각하면 좋아요.  
> **주의할 점** 나누는 수인 1/8만 뒤집어 곱해요.

### S07. 역질문과 힌트

#### 첫 역질문 카드

항상 힌트보다 먼저 강조한다.

```text
아이에게 먼저 이렇게 물어보세요
“이 문제는 3/4L를 몇 컵으로 나누는 걸까,
아니면 3/4L 안에 1/8L가 몇 번 들어가는지 묻는 걸까?”
```

보조 정보:

- 질문 의도
- 기대 신호
- 막혔을 때 이어 할 말

#### 힌트 카드

각 카드에는 다음만 포함한다.

- 단계명
- 부모가 말할 문장
- 이 단계의 목적
- 아이가 막히면 할 짧은 후속 문장

버튼:

- `아이가 아직 막혀 있어요 · 다음 힌트`
- `여기까지 이해했어요`
- `최종 풀이 보기`

“최종 풀이 보기”는 접근 가능하지만 주 버튼보다 시각적으로 덜 강조한다. 사용을 막는 죄책감 문구나 반복 확인은 쓰지 않는다.

## 4. 힌트 강도 정의

### 0단계. 역질문

목표: 아이가 문제 상황과 구해야 하는 것을 이해했는지 확인한다.

- 좋은 예: `한 컵의 양을 알고 있을 때 전체에서 컵 수를 구하려면 어떤 계산이 필요할까?`
- 나쁜 예: `3/4 ÷ 1/8은 얼마야?`

### 1단계. 방향 힌트

목표: 필요한 정보나 문제 구조를 찾게 한다.

- 답, 완성된 식, 핵심 계산 결과를 주지 않는다.
- 예: `전체 양 안에 한 컵의 양이 몇 번 들어가는지 생각해 보자.`

### 2단계. 개념·연산 힌트

목표: 사용할 개념 또는 연산을 선택하게 한다.

- 예: `“몇 번 들어가는지”를 구할 때는 나눗셈으로 나타낼 수 있어.`

### 3단계. 실행 힌트

목표: 첫 식 또는 바로 다음 행동을 제시한다.

- 예: `3/4 ÷ 1/8로 식을 세우고, 나누는 분수를 뒤집어 곱셈으로 바꿔 보자.`
- 가능한 경우 최종 계산 결과는 여전히 남겨 둔다.

### 최종 풀이

전체 식, 답, 이유, 검산을 제공한다.

## 5. 아이 반응에 따른 후속 멘트

AI 결과에는 다음 세 가지를 구분한다.

- `맞게 말했을 때`: 다음 단계로 이어 주는 짧은 칭찬
- `부분적으로 맞았을 때`: 맞는 부분을 먼저 확인하고 빠진 부분을 질문
- `막혔을 때`: 더 쉬운 표현이나 구체물 비유

권장 톤:

- `좋아, 무엇을 구하는지는 정확히 찾았어. 이제 어떤 계산인지 생각해 보자.`
- `여기까지는 맞아. 뒤의 분수를 왜 뒤집는지만 한 번 더 보자.`
- `괜찮아. 컵을 실제로 여러 개 놓는다고 생각해 보자.`

금지 톤:

- `이것도 모르니?`
- `아까 말했잖아.`
- `아주 쉬운 문제야.`
- `틀렸어. 다시.`

## 6. 최종 풀이 화면

구성 순서:

1. 최종 답
2. 핵심 식
3. 단계별 이유
4. 검산 또는 답의 타당성 확인
5. 부모가 마무리로 할 질문

마무리 질문 예:

- `왜 곱셈이 아니라 나눗셈을 했는지 설명해 볼래?`
- `답이 6컵이라는 게 문제 상황에서 말이 되는지 확인해 볼까?`

검증 상태 표시:

- `계산으로 확인했어요` (`verified`)
- `일부 계산을 확인했어요` (`partially_verified`)
- `자동 검산이 어려운 문제예요. 조건을 다시 확인해 주세요` (`unverified`)

## 7. 비슷한 문제 화면

문제를 먼저 보여 주고 답은 접는다.

구성:

- 새 문제
- `이 문제가 비슷한 이유`
- `힌트 보기`
- `풀이 보기`

생성 원칙:

- 같은 개념과 풀이 구조를 유지한다.
- 숫자는 계산 가능한 수준으로 바꾼다.
- 원문 고유명사와 문장을 복사하지 않는다.
- 원문이 그림 문제라면 MVP에서는 텍스트만으로 성립하는 대체 문제를 우선한다.
- 새 문제의 답과 풀이를 서버에서 먼저 검증한다.

## 8. 오류 및 지원 불가 문구

### 흐린 사진

`숫자나 기호를 정확히 읽기 어려워요. 문제에 가까이 가서 흔들리지 않게 다시 찍어 주세요.`

### 여러 문제

`문제가 여러 개 보여요. 도움받을 문제 하나만 잘라 주세요.`

### 그림 누락

`풀이에 필요한 그림이나 표가 잘린 것 같아요. 문제와 그림 전체가 보이게 다시 찍어 주세요.`

### 낮은 풀이 신뢰도

`문제는 읽었지만 풀이를 충분히 확인하지 못했어요. 숫자와 조건을 다시 확인하거나 다른 사진으로 시도해 주세요.`

### 범위 밖

`현재 버전은 초등 5~6학년 수학 한 문제를 중심으로 지원해요.`

## 9. 콘텐츠 길이 가이드

- 부모용 핵심 설명: 120자 안팎
- 역질문: 한 번에 1개, 최대 2문장
- 부모 멘트: 카드당 최대 3문장
- 힌트: 단계당 한 개념
- 최종 풀이: 보통 3~6단계
- 비슷한 문제: 한 문제만

길이 제한보다 의미와 자연스러움을 우선하되, 모바일 첫 화면을 긴 문단으로 채우지 않는다.

## 10. 용어

- 제품 내 `역질문`은 내부 기획 용어다. 사용자 화면에서는 `아이에게 먼저 물어볼 질문`처럼 자연스럽게 표현한다.
- `AI 신뢰도`보다 `다시 확인이 필요한 부분`을 우선 사용한다.
- `오답` 대신 `헷갈리기 쉬운 부분`을 기본 표현으로 사용한다.
