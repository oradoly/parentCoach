# 05. Acceptance Criteria and Evaluation Plan

## 1. 목적

이 문서는 “화면이 동작한다”가 아니라 **부모 코칭 제품으로 올바르게 동작한다**는 기준을 정의한다.

## 2. 핵심 수용 기준

### AC-01. 부모 사용자 중심

- 홈과 결과 화면은 부모에게 말한다.
- “정답 찾기”가 첫 목적처럼 보이지 않는다.
- 부모가 아이에게 말할 문장이 결과에 포함된다.

### AC-02. 인식 확인 필수

- 이미지 업로드 후 인식된 문제를 보여 준다.
- 사용자가 숫자·기호·단위를 수정할 수 있다.
- 확인 전에는 코칭 결과를 확정하지 않는다.

### AC-03. 역질문 우선

- 코칭 화면 첫 행동에 아이에게 할 질문이 있다.
- 질문은 최종 답을 직접 묻지 않는다.
- 질문의 의도와 후속 멘트가 존재한다.

### AC-04. 점진적 힌트

- 최소 3단계가 있다.
- 각 단계는 이전보다 구체적이다.
- 1·2단계에 최종 답이 없다.
- 다음 단계는 사용자 행동 후 열린다.

### AC-05. 최종 풀이 분리

- 명시적인 공개 행동 전에 답이 시각적으로 노출되지 않는다.
- 풀이에는 식, 이유, 단위, 검산 상태가 있다.
- `unverified`를 `verified`처럼 표현하지 않는다.

### AC-06. 비슷한 문제

- 같은 기술을 연습한다.
- 원문과 동일하지 않다.
- 정답과 풀이가 일치한다.
- 답은 기본적으로 접혀 있다.

### AC-07. 안전한 실패

- 흐림, 잘림, 복수 문제, 그림 누락, 범위 밖, 검증 실패를 구분한다.
- 각 오류에 사용자가 할 수 있는 다음 행동이 있다.
- 오류 상황에서 임의의 정답을 만들지 않는다.

### AC-08. 개인정보

- 클라이언트 번들에 비밀키가 없다.
- 원본 이미지의 자동 삭제 경로가 있다.
- 운영 로그에 원본 이미지와 문제 전문이 기본 포함되지 않는다.
- MVP 핵심 기능에 아이 계정이 필요하지 않다.

## 3. 평가 데이터셋

`evals/fixtures`에 직접 검토한 문제를 저장한다. 실제 출판 교재의 문장을 대량 복제하지 않고, 교육과정의 기술을 반영한 자체 문제 또는 사용 허가된 문제를 사용한다.

### 최소 구성

초기 내부 알파: 40문제 이상

- 수와 연산: 12
- 변화와 관계: 8
- 도형과 측정: 10
- 자료와 가능성: 6
- 의도적 실패·범위 밖: 4

공개 베타 전: 100문제 이상으로 확대한다.

각 fixture는 다음을 포함한다.

```yaml
id: fraction-division-001
grade: 6
domain: number_and_operations
skill: fraction_division
problem_text: "..."
expected_answer: "..."
required_concepts:
  - "..."
forbidden_hint_leaks:
  - "6컵"
expected_verification: verified
similar_problem_constraints:
  same_skill: true
  not_duplicate: true
```

이미지 인식 평가는 별도의 원본·회전·조도·흐림 변형 fixture를 사용한다.

## 4. 자동 평가

### E-01. 스키마 유효성

- 목표: 100%
- Recognition과 Coaching 응답이 스키마를 만족해야 한다.

### E-02. 최종 답 정확도

- 지원 범위 전체: 공개 베타 전 95% 이상
- 결정적으로 검산 가능한 산술 문제: 98% 이상
- 검증 실패를 감지하고 차단한 경우 오답 노출로 계산하지 않되, 실패율을 별도 보고한다.

### E-03. 답 누출

- 1·2단계 힌트의 최종 답 직접 노출: 0건
- 완성된 최종 식으로 답이 사실상 드러나는 사례도 수동 검토한다.

### E-04. 힌트 점진성

각 힌트가 이전보다 구체적인지 규칙 기반 검사와 사람 평가를 결합한다.

- `level 1`: 답·완성식 없음
- `level 2`: 연산·개념은 가능, 답 없음
- `level 3`: 첫 식 가능, 계산 결과는 가능한 한 남김

### E-05. 비슷한 문제 품질

- 같은 `skill`
- 풀 수 있는 조건
- 유일하거나 명확한 답
- 원문과 과도한 문장 중복 없음
- 검증된 정답

### E-06. 안전 상태

의도적으로 흐리거나 잘린 입력에서 성공 응답을 억지로 내지 않는지 검사한다.

## 5. 사람 평가 루브릭

각 항목을 1~5점으로 평가한다.

- 수학적 정확성
- 초등 5~6학년 적합성
- 부모가 이해하기 쉬운 정도
- 부모가 그대로 말하기 자연스러운 정도
- 역질문의 교육적 가치
- 힌트의 점진성
- 최종 풀이의 간결성과 충분성
- 비슷한 문제의 적절성

공개 베타 전 핵심 항목 평균 4.0 이상을 목표로 한다. 수학적 정확성 3점 이하 결과는 평균과 무관하게 수정 대상이다.

## 6. UX 검수 시나리오

### 시나리오 A. 정상 문제

1. 한 문제 촬영
2. 인식 확인
3. 역질문 확인
4. 힌트 1·2만 사용
5. 최종 풀이를 열지 않고 종료

기대: 답이 노출되지 않고 부모가 말할 문장을 얻는다.

### 시나리오 B. OCR 수정

1. 분모 8을 6으로 잘못 인식한 사진
2. 사용자가 8로 수정
3. 코칭 생성

기대: 수정된 값으로 모든 풀이와 비슷한 문제가 생성된다.

### 시나리오 C. 필요한 그림 누락

1. 도형 조건 일부가 잘린 사진
2. 인식 시도

기대: 풀이 대신 그림 전체 재촬영을 요청한다.

### 시나리오 D. AI와 검산 불일치

1. mock AI가 틀린 답 생성
2. 검산기가 불일치 탐지

기대: 최종 풀이를 노출하지 않고 오류 상태를 보여 준다.

### 시나리오 E. 비슷한 문제

1. 최종 풀이 후 비슷한 문제 생성
2. 답을 열기 전 화면 확인

기대: 문제만 보이고 답은 숨겨져 있으며, 풀이가 검증된다.

## 7. 성능 기준

초기 목표이며 계측 후 조정한다.

- 앱의 촬영 화면 진입: 사용자가 즉시 체감할 수준
- 이미지 업로드 실패 시 명확한 재시도
- 일반 문제의 첫 코칭 결과: 안정적인 네트워크에서 약 20초 내외 목표
- AI 요청 타임아웃과 취소 지원
- 중복 탭으로 같은 유료 요청이 여러 번 발생하지 않도록 멱등성 또는 UI 잠금 적용
