# 06. MVP Backlog

## 작업 원칙

- 각 마일스톤은 독립적으로 실행·검증 가능한 수직 기능을 만든다.
- 실제 AI 연결 전에 mock 데이터로 핵심 UX를 검증한다.
- 한 작업에서 제품 범위를 넓히지 않는다.
- 각 마일스톤 시작 전 `PLANS.md`를 갱신한다.

## M0. 저장소와 개발 기반

목표: 모든 후속 작업이 같은 규칙과 명령을 사용하게 한다.

작업:

- [ ] pnpm workspace 초기화
- [ ] `apps/mobile`, `apps/api`, `packages/contracts` 생성
- [ ] TypeScript strict, lint, format, test 구성
- [ ] 루트 명령 `dev`, `lint`, `typecheck`, `test`, `eval` 정의
- [ ] `.env.example`과 비밀 관리 규칙 추가
- [ ] CI에서 lint/typecheck/test 실행
- [ ] 문서 링크와 실제 명령 일치 여부 확인

완료 조건:

- 새 개발자가 README만으로 설치와 테스트를 실행할 수 있다.
- 빈 앱과 health API가 실행된다.
- 모든 검증 명령이 통과한다.

## M1. Mock 기반 부모 코칭 수직 흐름

목표: AI 없이 핵심 제품 경험을 먼저 검증한다.

작업:

- [ ] 홈, 촬영 placeholder, 인식 확인 화면
- [ ] mock 문제와 mock 코칭 응답
- [ ] 부모용 빠른 이해 카드
- [ ] 역질문 카드
- [ ] 3단계 힌트 점진 공개
- [ ] 명시적 최종 풀이 공개
- [ ] 비슷한 문제와 접힌 답
- [ ] 오류 상태 mock
- [ ] 컴포넌트·흐름 테스트

완료 조건:

- `docs/02_UX_AND_CONTENT.md`의 정상 흐름을 실제 기기에서 시연할 수 있다.
- 첫 화면과 1·2단계 힌트에 답이 없다.

## M2. 촬영·자르기·임시 업로드

목표: 실제 문제 사진 한 장을 안전하게 서버에 전달한다.

작업:

- [ ] 카메라와 사진 권한 처리
- [ ] 촬영, 선택, 회전, 자르기
- [ ] EXIF 위치 제거와 이미지 리사이즈
- [ ] 파일 크기·형식 검증
- [ ] 임시 세션과 업로드
- [ ] 업로드 취소·재시도
- [ ] TTL 삭제 및 삭제 테스트

완료 조건:

- 실제 휴대폰에서 사진을 올리고 세션 종료 후 삭제를 확인할 수 있다.
- 비밀키가 모바일 번들에 없다.

## M3. 문제 인식과 사용자 확인

목표: 이미지에서 문제를 읽고 사용자가 확정하게 한다.

작업:

- [ ] Recognition 공유 스키마
- [ ] AI 인식 어댑터
- [ ] Structured Output 적용
- [ ] 복수 문제·흐림·그림 누락 분기
- [ ] 낮은 신뢰도 구간 표시
- [ ] 텍스트·수식 수정 UI
- [ ] 수정 후 이전 결과 무효화
- [ ] 이미지 fixture 평가

완료 조건:

- 사용자가 확인한 문제만 코칭 입력으로 사용된다.
- 인식 실패에서 임의 풀이가 생성되지 않는다.

## M4. 코칭 생성

목표: 확인된 문제에서 부모용 코칭 계약을 만든다.

작업:

- [ ] Coaching 공유 스키마
- [ ] 제품 지침과 프롬프트 버전
- [ ] 부모용 빠른 이해
- [ ] 첫 역질문과 후속 멘트
- [ ] 3단계 힌트
- [ ] 최종 풀이
- [ ] 구조화 결과 UI 연결
- [ ] 스키마 실패·타임아웃·재시도

완료 조건:

- 40개 이상 fixture에서 스키마 유효성 100%
- 답 누출 검사 통과
- 부모용 문구 수동 검토 통과

## M5. 수학 검증 계층

목표: LLM의 그럴듯한 오답을 사용자에게 확정적으로 노출하지 않는다.

작업:

- [ ] 정확한 자연수·분수·소수 계산
- [ ] 단위 정규화와 기본 단위 검사
- [ ] 대입·역산 검산
- [ ] `verified` 상태 계약
- [ ] 생성 결과와 검산 불일치 차단
- [ ] 검증 실패 UX
- [ ] 관련 단위·통합 테스트

완료 조건:

- 결정적 검산이 가능한 fixture의 기준 정확도 충족
- mock 오답이 최종 풀이 화면에 노출되지 않는다.

## M6. 비슷한 문제 생성과 검증

목표: 한 문제를 푼 뒤 같은 개념을 확인하는 새 문제를 제공한다.

작업:

- [ ] 동일 skill·유사 난이도 제약
- [ ] 원문 중복 검사
- [ ] 새 문제의 정답·풀이 검산
- [ ] 문제 먼저, 답 접기 UI
- [ ] 생성 실패 시 안전한 대체 문구
- [ ] 평가 fixture

완료 조건:

- 생성된 문제마다 검증된 답이 있다.
- 원문 복제나 지나치게 어려운 숫자가 반복되지 않는다.

## M7. 개인정보·관찰 가능성·비용 통제

목표: 제한된 베타를 운영할 수 있는 기반을 만든다.

작업:

- [ ] 로그 마스킹
- [ ] 모델·프롬프트·스키마 버전 기록
- [ ] 지연·비용·오류 메트릭
- [ ] 속도 제한과 업로드 제한
- [ ] 세션 삭제 재시도와 모니터링
- [ ] 피드백·오류 신고
- [ ] 개인정보·이용 안내 초안
- [ ] 장애 시 kill switch 또는 모델 비활성화 플래그

완료 조건:

- 원본 이미지가 로그와 오류 도구에 포함되지 않는다.
- 비용 상한과 요청 제한이 동작한다.

## M8. 내부 알파와 공개 베타 준비

목표: 실제 부모 사용성으로 핵심 가설을 확인한다.

작업:

- [ ] 최소 40문제 자동·수동 평가 완료
- [ ] 부모 5~10명 내부 사용성 테스트
- [ ] 촬영부터 첫 질문까지 걸리는 시간 계측
- [ ] 이해하기 어려운 문구 개선
- [ ] 오답·인식 오류 분류
- [ ] 공개 베타 전 100문제 평가셋 확장 계획
- [ ] 출시 체크리스트 검토

완료 조건:

- `docs/05_ACCEPTANCE_AND_EVALS.md`의 공개 베타 기준을 충족하거나, 미충족 항목과 차단 결정을 명확히 기록한다.
